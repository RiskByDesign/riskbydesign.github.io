Security model

BowCRQ is a local-first application. This document states what that means precisely, for both the desktop app and the Android tablet build, so the guarantees are checkable rather than vibes.

Network

  • No telemetry, no analytics, no crash reporting, no update checks, no fonts or assets fetched at runtime. Everything ships in the binary.
  • The single permitted network call is the opt-in, bring-your-own-key AI review feature. The Content Security Policy restricts connect-src to the local IPC origin plus https://api.anthropic.com, so even that feature can only ever talk to one host, and only when a user has deliberately pasted their own API key.
  • The Android manifest requests INTERNET solely for that same opt-in feature and the WebView's local asset serving. There are no other permissions besides the biometric prompt.
  • The evidence register does not change this. Cited external sources are entered or imported by the analyst; the application never fetches them. A source's URL is stored as plain text and is not retrieved at runtime, and no third-party or industry data is bundled or downloaded. External context is cited, never live-fed.

Accounts and authentication

  • Accounts are local to the machine. No email, no cloud identity, nothing leaves the device.
  • Passwords are hashed with Argon2id. There is no recovery mechanism by design: a forgotten password means a lost account, stated in the UI.
  • Sessions are held in memory and end when the app closes.

Biometric unlock (Android, opt-in)

Biometric unlock never replaces or stores the password:

  • Only a successful password sign-in can mint a device session: a random 256-bit token whose secret is stored Argon2id-hashed in the local database, with a 30-day expiry and at most one session per user.
  • The raw token stays on the device and is inert until the operating system's biometric prompt succeeds, after which it is exchanged for the usual in-memory session.
  • Signing out revokes the session server-side and deletes the token. Password recovery semantics are unchanged: there still is none.
  • Desktop builds contain none of this; desktop sessions are never persisted.

Data at rest

The SQLite database lives in the per-user application data directory and is not separately encrypted. Confidentiality at rest is delegated to the platform: BitLocker or equivalent on Windows, file-based encryption on Android (universal since Android 10). The login gate controls access through the app, not through the filesystem; anyone with full filesystem access on an unlocked, unencrypted device can read the data. Use device encryption.

The observation and evidence registers store their contents in this same database: incident notes, cited sources, provenance and any data points the analyst records. This can be sensitive material, and it is protected exactly as the rest of the model is, by the platform's at-rest encryption. The loose export files these registers produce are plain JSON and warrant the same care as the database.

Change governance

Evidence-informed changes to a model are governed and recorded, not silent. When the analyst approves a calibration proposed from evidence, including a credibility-weighted blend of their own data with an external prior, the change is written to an append-only audit trail: the old and new value, the evidence and sources it drew on, the credibility weight where one applies, the analyst's mandatory rationale, and the local username. The trail lives in the local database with everything else. It is an integrity and accountability record for how the numbers were arrived at, not a security boundary, and like all other data it never leaves the device.

Android distribution

  • v1 distributes as a sideloaded APK signed with a release keystore that is held offline by the maintainer (see docs/android-signing.md; the keystore and its passwords are never committed).
  • The app enforces a minimum Android System WebView version (Chromium 110) with a clear upgrade message rather than rendering incorrectly.
  • Model files exported for the tablet-to-desktop hand-off are plain JSON, as documented; treat exported files with the same care as the database.

Reporting

This is a personal project without a bug bounty. Issues can be raised with the maintainer directly.
